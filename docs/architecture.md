
# Novelight 系统架构

## 架构总览

Novelight 不是一个聊天机器人，也不是一个日记 App。
它是一个**持续感知、持久理解、主动陪伴的 AI 自我理解系统**。

核心设计哲学：

> 用户的唯一任务是活着。Novelight 负责理解。

```
┌─────────────────────────────────────────────────────┐
│                      用户的生活                       │
│    对话 ┃ 聊天记录 ┃ 身体数据 ┃ 照片 ┃ 手写信        │
└────────────────────────┬────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│              第一层：自然流 (The Stream)               │
│                                                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │ 对话接口  │ │ 数据接入  │ │ 主动感知  │            │
│  │(对话Agent)│ │(采集Agent)│ │(守护Agent)│            │
│  └──────────┘ └──────────┘ └──────────┘            │
│                                                      │
│  职责：接收、时间戳标注、多模态处理、实时标注          │
```

## 第一层：自然流 (The Stream)

### 职责
接收用户生活的原始数据流，进行初步处理和标注，转化为可被河床消化的结构化事件。

### 三个 Agent

#### 1. 对话 Agent (Conversation Agent)

用户与 Novelight 交互的主要入口。

**功能：**
- 接收用户的自然语言输入（文字、语音）
- 在回应时查询河床获取用户的完整上下文
- 实时标注对话中的关键信息（事件、情绪、人物、触发点）
- 将标注后的对话事件写入河床

**关键特性：**
- 不是无状态的聊天——每一轮对话都建立在河床中积累的全部理解之上
- 用户感受到的是"它记得我"，技术上实现的是"它在每次回应前查询了关于我的整个知识图谱"

**技术实现方向：**
```
用户输入
    ↓
对话 Agent 接收
    ↓
查询河床 → 获取相关上下文（向量检索 + 图谱查询）
    ↓
构建 Prompt（用户输入 + 河床上下文 + 系统指令）
    ↓
调用 LLM 生成回应
    ↓
同时：从对话中提取结构化事件 → 写入河床
    ↓
返回回应给用户
```

#### 2. 采集 Agent (Ingestion Agent)

负责从外部数据源自动或半自动地接收数据。

**数据源：**

| 数据源 | 接入方式 | 频率 |
|--------|---------|------|
| 可穿戴设备（Apple Watch / 华为手环） | HealthKit / Health Connect API | 持续同步 |
| 聊天记录（微信等） | 用户授权导入（定期或一次性） | 用户触发 |
| 照片 | 用户上传 | 用户触发 |
| 日历/日程 | 日历 API | 持续同步 |
| 手写内容 | 拍照 + OCR | 用户触发 |

**处理流程：**
```
外部数据到达
    ↓
采集 Agent 接收
    ↓
数据清洗与标准化（时间戳对齐、格式统一）
    ↓
初步标注（数据类型、来源、时间范围）
    ↓
写入河床的原始数据层
    ↓
触发整合分析 Agent 进行跨源关联
```

#### 3. 守护 Agent (Guardian Agent)

Novelight 的"主动关怀"能力。不等用户来找它，它会在适当的时机找用户。

**触发条件与行为：**

| 触发信号 | 行为 |
|---------|------|
| 连续5天运动量骤降至日常水平的30%以下 | 温和询问："最近好像没怎么动，身体还好吗？" |
| 用户超过10天未与 Novelight 互动 | 轻触："有一阵子没聊了，想知道你最近过得怎样。" |
| 检测到已知模式再次出现 | 提示："你注意到了吗，上次类似的情况是在……" |
| 咨询预约前24小时 | 自动准备本周摘要并询问用户是否要补充 |
| 可穿戴数据中出现急性压力信号 | 提供即时支持资源 |
| 关键日期临近（如亲人离世周年） | 提前关怀 |

**关键限制：**
- 频率控制：不能变成骚扰。用户可以设定"免打扰"模式
- 语气校准：必须温和、非侵入性、永远给用户选择权
- 危机协议：检测到自杀/自伤信号时，升级响应——推送危机热线、建议联系紧急联系人

## 第二层：河床 (The Riverbed)

### 职责
储存、结构化、持续更新关于用户的理解。这是 Novelight 的核心智慧所在。

### 关键设计决策：理解独立于 LLM

```
             ┌─────────────┐
             │  任意 LLM    │ ← 可替换
             │ (GPT/Claude/ │
             │  Gemini/开源) │
             └──────┬──────┘
                    │ 查询 / 写入
                    ▼
        ┌───────────────────────┐
        │      河床（持久层）     │ ← 不可替换，永久存在
        │                       │
        │  用户拥有 · 可导出     │
        │  可检视 · 可修正       │
        └───────────────────────┘
```

LLM 是河床的"读者和书写者"，但河床本身不依赖任何特定的 LLM。如果 OpenAI 明天关闭了 API，用户的理解完好无损——换一个模型接入同一个河床，继续工作。

### 河床的四个子系统

#### 2.1 事件图谱 (Event Graph)

**存储方式：** 知识图谱（如 Neo4j）

**节点类型：**
- 事件节点（Event）：发生了什么，时间戳
- 人物节点（Person）：涉及谁
- 情绪节点（Emotion）：当时的感受
- 主题节点（Theme）：属于什么领域（家庭、关系、工作、健康……）
- 身体数据节点（BodyData）：关联的生理数据

**边类型：**
- `TRIGGERED_BY`：这个事件被什么触发
- `LED_TO`：这个事件导致了什么
- `SIMILAR_TO`：这个事件和哪个过去的事件结构相似
- `INVOLVES`：涉及哪些人物
- `FELT`：当时的情绪
- `CORRELATED_WITH`：和哪个身体数据关联

**示例：**
```
[事件: 深夜独自改稿第四版 (2026-04-02)]
    ──TRIGGERED_BY──> [模式: 客户含糊反馈后的无限返工]
    ──SIMILAR_TO──> [事件: 年初为甲方反复改logo (2026-01-20)]
    ──SIMILAR_TO──> [事件: 给导师交论文被退回三次 (2025-11-12)]
    ──FELT──> [情绪: 自我怀疑、耗竭]
    ──CORRELATED_WITH──> [身体数据: 连续4晚睡眠不足5小时]
    ──LED_TO──> [事件: 第二天向朋友小柚倾诉 ]
    ──LED_TO──> [事件: 尝试设定交付边界 ]
```

#### 2.2 模式库 (Pattern Library)

**存储方式：** 结构化文档（JSON/YAML）+ 向量索引

**模式的数据结构：**
```yaml
pattern:
  id: "PTN-012"
  name: "模糊评价下的完美主义螺旋"
  category: "工作-自我价值"
  first_observed: "2025-11-12"
  description: >
    当外部权威（客户、导师、长辈）给出模糊或
    非具体的负面评价时，用户会陷入过度修改循环，
    将"不够好"等同于"我整个人不行"。
  trigger_conditions:
    - "客户/上级说'再调调'但不给具体方向"
    - "收到'还可以吧'之类的敷衍认可"
    - "看到同行作品获得大量好评"
  typical_response_chain:
    - "反复检查自己的作品找问题"
    - "→ 推翻重来（即使已经很好）"
    - "→ 熬夜赶工 → 身体透支"
    - "→ 交付后仍焦虑等待评价"
    - "→ 获得认可后短暂释然 → 下一次循环"
  occurrences:
    - date: "2025-11-12"
      severity: 9
      recovery_time: "weeks"
      notes: "毕设论文被导师退回。连续一周失眠，怀疑自己不适合做设计。"
    - date: "2026-01-20"
      severity: 8
      recovery_time: "5 days"
      notes: "甲方说logo'没感觉'。推翻4版，最终对方选了第1版。"
    - date: "2026-03-10"
      severity: 6
      recovery_time: "2 days"
      notes: "朋友转发竞品设计获奖。整夜对比自己作品。小柚提醒后平复。"
    - date: "2026-04-02"
      severity: 4
      recovery_time: "3 hours"
      notes: "客户要求改第四版。主动问清具体方向后冷静执行。"
  evolution_trajectory: "严重度递减，能越来越快地识别'我又在螺旋了'。开始学会主动索要具体反馈。"
  current_status: "改善中。最近一个月未出现完全推翻重来的行为。"
  related_trauma: "小学画展作品被父亲当众评价'画得什么东西'；高中被美术老师忽视"
  coping_strategies_developed:
    - "交稿前先发给小柚做'现实核查'"
    - "建立'三版封顶'规则——超过三版必须要求对方给明确方向"
    - "记录每次获得好评的截图放进'证据文件夹'"
```

#### 2.4 进化轨迹 (Evolution Tracker)

**存储方式：** 时间序列数据 + 结构化里程碑

这是河床中对咨询师最有价值的组件。

```yaml
evolution:
  dimension: "职业自我效能感"
  milestones:
    - date: "2023-06"
      state: "完全依赖外部评价确认自己的能力"
      evidence: "每份作品都要发给三个人确认才敢提交"

    - date: "2024-01"
      state: "开始接自由职业项目但定价极低"
      evidence: "第一个私单报价只有市场价的40%，怕别人觉得不值"

    - date: "2025-03"
      state: "敢提出合理报价但被砍价就立刻妥协"
      evidence: "客户说'贵了'，马上打七折并免费加一轮修改"

    - date: "2025-11"
      state: "论文退回触发最严重的一次自我怀疑"
      evidence: "连续一周失眠，在日记里写'也许我根本没有才华'"

    - date: "2026-01"
      state: "logo反复修改事件后开始意识到模式"
      evidence: "在和小柚的对话中第一次说出'我好像又在讨好了'"

    - date: "2026-03"
      state: "能在被触发时暂停而不是立刻行动"
      evidence: "看到竞品获奖，没有连夜改自己的作品，而是先睡觉"

    - date: "2026-04"
      state: "主动向客户索要明确反馈而非猜测"
      evidence: "'你说再调调，能告诉我具体哪里不满意吗？'"

  overall_trajectory: >
    从完全依赖外部评价到逐步建立内在标准。
    两年内完成了从"别人说好才是好"到
    "我知道这个作品是好的"的初步转化。关键催化剂：
    小柚的持续无条件支持 + 自由职业中被迫面对真实市场反馈 + AI辅助的模式觉察。
```

### 整合分析 Agent (Integration Agent)
河床里的"后台大脑"。不面向用户，在后台持续工作。

**职责：**
- **跨源关联：** 当新的对话数据和新的身体数据同时到达时，自动检查两者之间是否存在有意义的关联。比如："用户今天提到和妈妈吵架了" + "今晚睡眠数据异常差" → 标注关联
- **模式发现：** 当某个行为第三次出现时，自动检测并生成模式草稿，推送给用户确认
- **画像更新：** 当累积了足够的新数据时，更新用户的人格画像和进化轨迹
- **异常检测：** 检测偏离用户正常模式的信号（突然的情绪崩溃、异常的睡眠、长时间的沉默）

**运行频率：**
- 实时：对话事件的处理和标注
- 每日：身体数据的关联分析
- 每周：模式库的扫描和更新
- 每月：画像和进化轨迹的整体更新

## 第三层：书 (The Book)

### 职责
将河床中的结构化理解渲染成人类可阅读的叙事，分别面向用户（C端）和咨询师（B端）。

### 叙事生成 Agent (Narrative Agent)

**输入：** 河床中特定时间范围的结构化理解
**输出：** 有章节结构的叙事文本

**生成流程：**
```
触发条件（每周/每月/用户请求/重大事件后）
    ↓
叙事 Agent 查询河床
    ↓
获取时间范围内的：事件、模式、画像变化、身体数据
    ↓
构建叙事 Prompt
    ↓
调用 LLM 生成章节草稿
    ↓
用户审阅 → 确认 / 修改 / 补充
    ↓
定稿存入书的章节库
```

**叙事质量要求：**
- 不是日志摘要，是故事
- 好的叙事示例：*"这是她终于不再为自己的需求感到羞耻的那个月。"*
- 坏的叙事示例：*"用户在3月份与伴侣确认了关系并解决了多次沟通冲突。"*

### C端界面

用户看到的是一本书。

**核心页面：**
- **书架页：** 按时间/主题/关系组织的章节列表
- **章节页：** 叙事摘要 + 锚点（可展开看原始素材）
- **对话页：** 和 Novelight 聊天的入口
- **模式页：** 已识别的模式列表，每个可展开看详细记录和进化轨迹
- **我的画像页：** 当前的自我画像，可以看历史版本的变化

### B端界面

咨询师看到的是一个经过筛选的、结构化的、带有导航的视图。

**核心页面：**
- **来访者速写：** 一页纸的关键信息——核心议题、依恋类型、主要触发点、当前状态
- **章节目录：** 来访者人生/近期的关键阶段，可快速跳转
- **模式报告：** 已识别的模式，附带频率、严重度、进化趋势的可视化
- **进化轨迹：** 来访者在咨询前和咨询中的变化，用时间线可视化
- **原始素材浏览器：** 用户授权开放的对话片段、身体数据、手写内容

### 隐私与权限控制层

贯穿整个第三层的底座。

**权限模型：**
```
每一个数据单元（章节、模式、对话片段、身体数据……）
    ↓
    有三种可见性设置：
    ├── 仅自己可见（默认）
    ├── 对特定咨询师可见（用户显式授权）
    └── 对研究匿名可见（用户显式同意）
```

**用户操作：**
- 可以随时修改任何数据的可见性
- 可以一键撤销对某个咨询师的所有授权
- 可以导出所有数据
- 可以删除任何数据（包括河床中的衍生理解）

## 数据流总览

```
用户说了一句话
    ↓
对话 Agent 接收
    ↓
查询河床 → 发现这和三个月前的一个事件结构相似
    ↓
在回应中提及这个关联："你注意到了吗，上次类似的情况是……"
    ↓
同时：提取结构化事件 → 写入事件图谱
    ↓
整合分析 Agent 检测到：这是某个模式的第四次出现
    ↓
更新模式库中的该模式记录
    ↓
触发守护 Agent：推送模式提醒给用户
    ↓
周末：叙事 Agent 将本周的事件整合成章节草稿
    ↓
用户确认章节 → 存入书
    ↓
```

## 与 Agent 生态的关系

Novelight 不是一个通用 Agent 框架。它是一个**垂直领域的 Agent 系统**，专注于"理解人"这一件事。

与通用 Agent 的区别：

| 维度 | 通用 Agent | Novelight |
|------|-----------|-----------|
| 核心能力 | 执行任务（搜索、编码、调用工具） | 理解人（感知、记忆、关联、叙述） |
| 记忆系统 | 通常是向量检索（片段式） | 结构化知识图谱 + 模式库 + 进化轨迹（整体式） |
| 主动性 | 按指令行动 | 基于对用户的理解主动关怀 |
| 时间尺度 | 单任务/单会话 | 跨越数月到数年 |
| 核心产出 | 完成任务的结果 | 一本关于你的、活着的书 |

Novelight 可以在 Agent 生态中定位为：**最懂用户的那个 Agent**。其他 Agent 做任务，Novelight 做理解。未来甚至可以作为其他 Agent 的"用户上下文供应商"——当一个健康管理 Agent 需要知道"这个用户为什么最近睡不好"时，它可以（在用户授权下）查询 Novelight 的河床。
