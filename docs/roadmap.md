# Novelight 产品路线图

## 版本规划总览

---

## Phase 0：奠基

**目标：** 确定核心架构，搭建最小可运行原型。

### 0.1 架构与技术选型

- [ ] 确定持久理解层的技术栈
  - 知识图谱：Neo4j vs 轻量级替代（如 TypeDB）
  - 向量存储：Pinecone / Weaviate / Qdrant / 自建
  - 结构化摘要存储：PostgreSQL + JSON
- [ ] 确定 LLM 接入策略
  - 初期：Claude API + GPT API（双路由，互为备份）
  - 计划：开源模型（如 Llama / Qwen）本地部署能力
- [ ] 确定数据加密方案
- [ ] 搭建基础开发环境

### 0.2 核心验证实验

- [ ] 用创始人自身数据（已有几万字的对话记录和分析）搭建第一个河床原型
- [ ] 验证：从结构化河床数据出发，LLM 能否在新会话中"认识"用户？
- [ ] 验证：跨会话的模式检测是否可行？
- [ ] 验证：叙事生成的质量是否达到"读起来像故事而不是摘要"的标准？

### 0.3 初始团队组建

- [ ] 明确创始人分工
  - Flewolf：产品、叙事设计、用户体验、AI交互策略
  - Snowolf：心理学框架、临床安全性、技术实现
- [ ] 识别需要补充的角色（前端、后端、知识图谱工程）

**里程碑：** 一个能在命令行里运行的、基于创始人数据的"河床 + 对话"原型。用户输入一句话，系统查询河床返回带上下文的回应。

## Phase 1："它记得我" (MVP)

**目标：** 第一个可以给真实用户使用的版本。核心体验是"和一个记得你的AI对话"。

### 1.1 对话系统

- [ ] 带持久记忆的对话界面
- [ ] 每次回应前自动查询河床获取相关上下文
- [ ] 对话中的自动事件提取（人物、情绪、主题、时间）
- [ ] 跨会话连续性——用户隔了一周回来，AI记得上次聊到哪里

### 1.2 河床 v1

- [ ] 事件图谱 v1：基础的事件节点和关联
- [ ] 画像 v1：从对话中自动提取的用户基本信息和初步特征
- [ ] 简单的模式检测：当同一主题/情绪第三次出现时标记

### 1.3 可穿戴数据接入 v1

- [ ] Apple HealthKit 基础接入（睡眠、心率）
- [ ] 数据和对话的时间关联（"你昨天提到压力很大，你的睡眠数据也显示昨晚深睡只有48分钟"）

### 1.4 基础叙事

- [ ] 每周自动生成一段"本周摘要"
- [ ] 用户可以查看和修改

### 1.5 隐私基础

- [ ] 用户账号系统
- [ ] 数据加密存储
- [ ] 数据导出功能
- [ ] 数据删除功能

**里程碑：** 20-50个种子用户（可以从创伤恢复社区、心理学学生中招募）使用2个月。核心验证指标：
- 留存率（用户是否持续回来对话？）
- "它记得我"的主观感受评分
- 用户是否自发地分享更深层的内容？

---

## Phase 2："它看见我的模式" (核心差异化)

**目标：** 从"记得你说了什么"升级到"看见你行为/情绪/关系中反复出现的模式"。这是 Novelight 和所有其他AI对话产品的本质区别。

### 2.1 模式检测引擎

- [ ] 时序模式检测：从事件图谱中自动发现反复出现的行为序列
- [ ] 触发条件识别：什么情况下某个模式会被激活
- [ ] 严重度评估：每次出现的强度
- [ ] 进化追踪：同一个模式随时间是变好还是变坏

### 2.2 河床 v2

- [ ] 完整的模式库
- [ ] 进化轨迹追踪
- [ ] 多源数据关联（对话情绪 × 身体数据 × 事件时间线）
- [ ] 人物关系图谱（用户和重要他人之间的关系动力学）

### 2.3 守护 Agent v1

- [ ] 基于模式的主动提醒
- [ ] 基于身体数据异常的主动询问
- [ ] 咨询预约前的自动摘要准备

### 2.4 深度分析能力

- [ ] 用户可以请求"帮我分析这段关系"、"帮我画心理地图"
- [ ] 分析基于河床的完整上下文，不需要用户重新提供背景
- [ ] 分析结果写回河床，持续积累

**里程碑：** 用户开始主动说"Novelight 帮我看到了我自己没注意到的东西"。核心验证指标：
- 模式识别的准确率（用户确认vs否认）
- "洞察感"评分——用户是否觉得得到了新的自我理解？
- 进化轨迹是否和用户的主观感受一致？

---

## Phase 3："它讲述我的故事" (C端完整体验)

**目标：** 实现完整的"书"体验。用户打开 Novelight 看到的不再只是对话记录，而是一本有章节的、关于自己人生的书。

### 3.1 叙事生成 v2

- [ ] 完整的章节结构（自动根据转折点划分）
- [ ] 叙事摘要 + 锚点 + 支撑素材的三层结构
- [ ] 空白时段的叙事处理
- [ ] 用户可以编辑和补充叙事

### 3.2 书的UI

- [ ] 书架页——按时间/主题/关系浏览
- [ ] 章节页——沉浸式的叙事阅读体验
- [ ] 锚点展开——点击锚点查看原始对话/数据/照片
- [ ] 进化轨迹可视化——用户可以直观看到自己的变化曲线

### 3.3 回溯性标注

- [ ] 用户可以回到过去的事件上添加"现在的理解"
- [ ] "当时我以为是X，现在我知道其实是Y"——这种认知的进化本身被记录

### 3.4 高级数据接入

- [ ] 聊天记录批量导入（微信、QQ等，通过开源工具）
- [ ] 照片时间线整合
- [ ] 日历/行程数据

**里程碑：** 用户在和咨询师的会谈中主动说"我可以给你看我的Novelight"。

---

## Phase 4："它让咨询更好" (B端)

**目标：** 为心理咨询师提供结构化的来访者理解工具。

### 4.1 咨询师端界面

- [ ] 来访者列表与管理
- [ ] 一页纸速写自动生成
- [ ] 章节目录导航
- [ ] 模式报告（带数据可视化）
- [ ] 进化轨迹时间线
- [ ] 原始素材浏览器（基于用户授权）

### 4.2 咨询师标注层

- [ ] 咨询师可以在来访者的书中添加专业标注
- [ ] 标注经来访者同意后整合进书中
- [ ] 咨询师的标注不会被来访者看到除非双方同意

### 4.3 授权与共享系统

- [ ] 来访者精细化控制：哪些章节/模式/素材对哪个咨询师可见
- [ ] 授权可随时撤销
- [ ] 授权审计日志（谁在什么时候看了什么）

### 4.4 会谈整合

- [ ] 咨询前：自动准备"本周/本月发生了什么"的摘要
- [ ] 咨询后：来访者可以记录"这次会谈的收获/感受"，整合进书中

**里程碑：** 10个咨询师使用3个月后的反馈收集。核心验证指标：
- 咨询师报告的"了解来访者"的速度是否提升？
- 来访者报告的咨询满意度是否提升？
- 咨询师是否愿意付费？

---

## Phase 5：生态 (开放平台)

**目标：** 从独立产品走向平台。

### 5.1 开放API

- [ ] 第三方健康工具可以（在用户授权下）查询 Novelight 的理解层
- [ ] 示例：一个冥想App可以根据 Novelight 知道的用户当前压力来源，推荐针对性的冥想内容

### 5.2 咨询平台对接

- [ ] 与简单心理、壹心理等平台的对接
- [ ] 来访者在平台上预约咨询师时，可以选择分享 Novelight 数据

### 5.3 研究合作

- [ ] 匿名化、知情同意的数据集
- [ ] 与高校和研究机构合作，用 Novelight 数据推进依恋、创伤、关系动力学等领域的研究
- [ ] 发表研究论文

### 5.4 多语言与多文化

- [ ] 英文版本
- [ ] 文化敏感的叙事生成（不同文化背景下同一行为的不同解读）
- [ ] 多语言的咨询师端

### 5.5 青少年版本

- [ ] 针对15-18岁用户的特殊设计
- [ ] 更强的安全保护
- [ ] 监护人角色（类似咨询师的有限可见性）
- [ ] 学校心理辅导老师的接入

## 时间线估算（高度不确定，仅供参考）

| 阶段 | 预估时间 | 关键产出 |
|------|---------|---------|
| Phase 0 | 1-2个月 | 架构确定、原型、团队 |
| Phase 1 | 3-4个月 | MVP上线、种子用户 |
| Phase 2 | 3-4个月 | 模式检测、守护Agent |
| Phase 3 | 3-4个月 | 完整的书体验 |
| Phase 4 | 4-6个月 | B端上线、咨询师合作 |
| Phase 5 | 持续 | 生态建设 |

**总计到B端可用：约14-20个月。**

但这不是线性的。Phase 1 和 Phase 2 的验证结果可能导致方向调整。Phase 4 的咨询师反馈可能回溯影响 Phase 2 和 3 的设计。保持灵活。

## 关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM API 成本过高 | 无法持续运营 | 开源模型兜底 + 分级调用策略（简单查询用小模型，深度分析用大模型） |
| 用户隐私泄露 | 信任崩塌 | 端到端加密 + 零知识架构 + 最小权限原则 |
| 模式识别不准确 | 用户失去信任 | 透明度原则——所有推断都附带"为什么"和"你觉得对吗？" |
| 用户沉迷AI分析而不去做咨询 | 延误真正需要专业帮助的用户 | 系统主动推荐咨询 + 明确"Novelight不是咨询师" |
| B端咨询师不接受 | 收入模型受阻 | 从愿意尝试新工具的年轻咨询师入手 + 用疗效数据说话 |
| 监管风险（医疗/数据） | 合规问题 | 明确定位为"自我理解工具"而非"医疗设备" + 法律顾问 |
